- A100 GPU is 3–5x faster than RTX 6000 GPU (2039 vs 672 GB/s memory bandwidth)
- A100 CPU is 3x faster than RTX 6000 CPU (64-core EPYC vs Xeon)
- GPU-over-CPU speedup increases on A100 (3–5x vs 2–3x on RTX 6000) because GPU bandwidth advantage grows faster than CPU core scaling

## OOM Boundary

GPU working pool (18 GB) is the bottleneck. Q04 (daily aggregation with ORDER BY) OOMs first at 619M rows; SANKEY (3-hop with UNION ALL) at 672M rows; all queries at 770M rows.

## Footnotes

¹ **Warm-cache assumption**: All GPU timings use pre-loaded GPU buffers (`gpu_buffer_init`). In the expected TRM deployment, transaction data is loaded once and Sirius serves as an analytical layer over resident data — matching this warm-cache scenario. Cold-start (first query after data load) adds a one-time transfer cost proportional to table size; subsequent queries on the same tables pay no transfer penalty.

² **Dictionary encoding**: All join keys and group-by columns are integer-encoded. The raw `entity_address_map` table uses VARCHAR entity names; we pre-compute `entity_address_map_int` with `entity_id BIGINT` + `addr_id INT` (322 distinct entities, 3.2M rows). Similarly, the `asset` VARCHAR column is replaced by `asset_id INT` via a dictionary table. This is standard database normalization — a one-time ETL step, not a query-time cost. The impact is significant: VARCHAR hash joins are not supported on GPU and fall back to CPU; with integer keys, the same queries run entirely on GPU. On the RTX 6000, Q02 with VARCHAR `asset` column took ~3.3s on GPU (memory spilling) vs 43ms with integer `asset_id` — a ~75x difference.

³ **String operation limitations**: GPU-accelerated hash joins require integer or fixed-width key types. Queries with VARCHAR join keys (DuckDB internal type 12) fall back to CPU execution. This is a fundamental constraint of the current cuDF-based join implementation, not a data modeling issue. The workaround (integer dictionary encoding) is transparent to the application layer and reduces both memory footprint and join cost.

⁴ **gpu_execution path**: These results use the `gpu_processing` (legacy) code path. The newer `gpu_execution` path offers table-level GPU caching (`scan_cache_level = 'table_gpu'`) but currently has a cache deadlock bug under investigation. Early gpu_execution results on Q03 show 122ms vs 38ms for gpu_processing — the 84ms overhead gap is from plan translation and memory management layers not yet optimized. This is an active area of development.
